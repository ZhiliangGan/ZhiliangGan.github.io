1. 项目概览

QMT 是一个面向个人/私募量化交易者的 A 股量化交易全流程系统，覆盖「策略研究 → 批量回测评估 → 盘中监控 → 实盘交易执行」四个阶段。

系统以 Python Monorepo 形式组织，由多个子模块构成，共享同一套配置管理与数据层。其中两个模块（config 和 qdata）通过 pip install -e . 安装为本地 Python 包，可被其他模块用 import 直接引用；另外三个（qss、sps、SAF）是直接通过命令行运行的应用程序，不需要安装。

技术栈为 Python 3.10+，核心中间件包括：QMT xtdata/xttrader（Windows 实盘）、AKShare（跨平台行情备份）、FastAPI（监控 API）、SQLite（历史数据本地缓存）、LLM API（辩论决策引擎）。

系统支持 Windows（实盘）/ Mac+Linux（研究、回测、监控）双平台运行，数据源在不可用时自动降级。

2.2 各层/模块职责

config/ — 统一配置中心

  • 文件：base.yaml、gateway.yaml、llm.yaml、strategies.yaml、watch_list.yaml、golden_pool.yaml
  • 职责：管理系统级参数（数据路径、日志、数据源优先级、回测默认参数、风控阈值、策略列表、通知 Webhook）
  • 入口：loader.py，被 QSS/SPS/SAF 通过 from config import ConfigManager, get_settings 导入

qdata/ — 统一数据网关（跨平台）

  • qdata/qdata/adapters/：多数据源适配器（12个文件），按优先级支持：
    • qmt_adapter.py — QMT xtdata（Windows，最高优先级）
    • qmt_remote_adapter.py — gRPC 远程 QMT
    • akshare_adapter.py — AKShare（跨平台，免费）
    • csv_adapter.py — 本地 CSV
    • tdx_api_adapter.py — 通达信 TdxClaw HTTP API
    • tdx_etf_adapter.py、tdx_futures_adapter.py、tdx_cb_adapter.py 等专用适配器
  • qdata/qmt_gateway/：gRPC 桥接服务，在 Windows 宿主运行，Mac/Linux 通过 gRPC Remote 访问 xtdata
  • qdata/service/data_service.py：DataService 统一服务入口，初始化时按 prefer 参数决定适配器顺序（默认 auto：qmt_remote → tdx_api → qmt → akshare → csv），检测每个适配器可用性，维护 available_sources 列表。load_bars() 方法先查本地 SQLite 缓存，未命中再按优先级降级拉取，拉到后写缓存。
  • qdata/storage/bar_store.py：存储层使用 SQLite（bars.db），表结构：bars(symbol, frequency, datetime, open, high, low, close, volume, turnover, adj_factor)，支持 UPSERT 增量合并。

qss/ — 实盘交易执行系统

  • 入口：main.py
  • 子模块：
    • core/data_feed.py — 行情统一接口
    • core/event_bus.py — 异步事件总线
    • core/signal_processor.py — 信号消费者（风控验证 → 执行）
    • core/notification.py — 飞书通知桥
    • strategy/ — 策略实现（MACross、PEValue、Trend、Timing、Resonance）
    • executor/ — 执行器（qmt_executor、simulated、remote）
    • risk/ — 风控（risk_manager、stop_loss、portfolio_provider）
    • llm/client.py — AI 客户端
    • api/ — FastAPI Web Dashboard

sps/ — 盘中信号监控系统

  • 入口：wave_monitor.py（直接运行）
  • 四层分离架构：data_adapter → async_fetcher → vector_engine → wave_monitor
  • 状态持久化：.wave_status_db.json、.wave_signal_cache.json
  • 数据库：wave_history.db（SQLite）

SAF/ — 策略评估工厂

  • 实际路径：SAF/strategy_factory/（核心执行目录）
  • 入口：run_evaluate.py（3891 行，批量评估主控）
  • 新 SAF 引擎核心（SAF/core/saf/）已迁移完成（Phase 1 ✅）

its/ — LLM 辩论决策引擎

  • debate/engine.py — DebateEngine（维度隔离辩论）

补充说明 ③ — TDX API（通达信 TdxClaw HTTP API）

SAF 是策略研究枢纽，承担「策略 JSON → 自动批量回测 → 打分排名」流水线。

6维评分系统

维度权重核心指标满分条件
收益（Return）35%CAGR(40%) + Sharpe(30%) + PF(30%)CAGR≥50%、Sharpe≥3、PF≥3
稳健性（Robust）45%MaxDD(30%) + Calmar(25%) + 滚动Sharpe(25%) + Sortino(20%)MaxDD<0%、Calmar≥3
可实施性20%胜率(30%) + 赔率(30%) + 持仓天数(20%) + 换手率(20%)胜率≥70%、赔率≥3
风控（独立）—MaxDDMaxDD<0%
效率（独立）—Calmar + SortinoCalmar≥3、Sortino≥4
一致性（独立）—滚动Sharpe标准差波动越小越好
总分 = 0.35×收益 + 0.45×稳健性 + 0.20×可实施性（稳健性权重最高）

硬淘汰规则（任一触发 → 直接 eliminated）

  • 最大回撤 > 40%
  • 夏普 < 0.3
  • 年化收益 < 0%（扣费后）
  • 交易次数 < 6 次
  • 胜率 < 20%

Tier 分级

S ≥85分  A ≥70分  B ≥55分  C <55分  eliminated

策略触发机制

StrategyBase.start()
 ├─ on_init() ← 子类覆写，加载历史数据等
 ├─ _bind_events() ← 关键：EventBus 事件绑定
     ├─ event_bus.subscribe(EVENT_TICK, self._handle_tick)
     ├─ event_bus.subscribe(EVENT_BAR, self._handle_bar)
     ├─ ...
 └─ on_start() ← 子类覆写，订阅行情或启动定时器

信号处理链

Strategy.on_bar() → event_bus.publish(EVENT_SIGNAL, TradeSignal)
  → SignalProcessor._handle_signal()
     ├─ _validate_signal()
         ├─ 检查 executor.connected
         ├─ 检查 signal.confidence >= 0.5
         └─ risk_manager.check_signal()
     ├─ [approved] _execute_signal() → executor.order_stock()
     └─ event_bus.publish_async(EVENT_SIGNAL_VALIDATED / REJECTED)
重要业务规则：confidence < 0.5 → 信号直接拒绝；executor 未连接 → 信号直接拒绝；dry_run=True → 检查 QMT 客户端但仍可运行。

场景 D — LLM 辩论决策

入口：its/screener.py → StockScreener.screen_symbol()

DebateEngine.run_debate(symbol, context)
 ├─ _build_dimension_context("technical") → 仅技术指标
 ├─ _build_dimension_context("fundamental") → 仅基本面
 ├─ _build_dimension_context("sentiment") → 仅情绪

 ├─ [if fundamental] _run_fundamental_agent() → FundamentalOutput

维度隔离设计：每个 Agent 的 prompt 只包含自身域数据，Arbiter 只看三个 Agent 的结论摘要，不直视原始数据。
signal_adapter.py 转换规则：
 • arbiter.action ∈ {BUY, SELL, CLOSE} 且 confidence ≥ 0.6 → 产生 TradeSignal
 • arbiter.action ∈ {HOLD, WATCH} 或 confidence < 0.6 → 返回 None
 • BUY 使用 context.technical.price；SELL 使用 arbiter.stop_loss_price
 • 产生的 TradeSignal 同样走 EventBus 流程，进入 SignalProcessor

Q5：评分标准是否偏高？真实通过率低的原因

结论：评分标准不高，是输入策略质量差和场景不匹配导致低通过率

证据 1 — 唯一通过 Stage2 的「RSRS择时结合PB-ROE」策略，2021-2024 连续三年亏损，靠 2025 年 +46.23% 才拉正全局 CAGR 到 15.61%，Sharpe 仅 0.48。这样的策略仍能 C 级通过，说明门槛不严。

证据 2 — Stage1 失败原因不是评分：no_signals（策略不产生信号）、signal_exception（抛异常）、too_few_trades（交易次数不足）。

证据 3 — 演进日志明确方向是「提升 Stage1 通过率（目标 30%+）」，而非降低 Stage2 评分标准。

Q6：配对交易策略失败根因与修复

失败表现

评估结果：status: failed，tier: failed，所有测试标的报 signal_func_error

三层根因

  1. 代码错误：df['close'].iloc[:, 1] — 单列 Series 无第二列，IndexingError
  2. 路由错误：kind 应为 pairs，但被路由到 TimeSeriesExecutionEngine
  3. 参数失效：beta=0.0567 异常偏小（应为 1-4）；p_mean/p_std 为静态值

v2 修复版回测结果（601988 + 002048，2.77年）

指标值说明
总收益率14.97%
CAGR5.16%偏低但为正，可进 Stage2
MaxDD19.52%中等

修复核心代码对比

# v1 原始（❌ 报错）
def generate_signals(df):
    portf = df['close'].iloc[:, 0] - beta * df['close'].iloc[:, 1]

# v2 修复（✅ 正常运行）
def generate_signals(df_a, df_b, config=None):
    # 动态 OLS 估计 beta
    beta = _estimate_beta_ols(close_a.iloc[:train_days], close_b.iloc[:train_days])
    spread = close_a - beta * close_b
    # 滚动窗口 z-score
    zscore = _rolling_zscore(spread, roll_window=60)
    signal[zscore < -1.5] = 1
    signal[zscore > +1.5] = -1
    return signal

7.3 数据缺口详细清单

无论传入什么指数代码，返回整个 universe_symbols，77 个策略选股范围与设计预期完全不同。
